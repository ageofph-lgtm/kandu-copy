KANDU — Fluxograma do app estado atual

Marketplace de construção (Base44 + React/Vite, espelhado em Supabase). Profissionais worker ↔ empregadores employer, com painel admin.
Worker Employer Admin Estado da obra Backend / função Dados / Supabase

Visão geral & mapa de navegação

Entrada por Welcome → login Base44 → SetupProfile → Home. O gating é feito no AuthContext/Layout. Worker e employer partilham o menu — a diferenciação acontece dentro das páginas. Admin é redirecionado de Home para o AdminDashboard.

Onboarding (Welcome → SetupProfile → Home)

Escolha de perfil, consentimento GDPR e verificação de identidade opcional. Employer tem passo extra de subtipo (simple/cia). Documento de ID dá verified_level: ultra_verified e +100 XP; sem documento fica verified. Admin salta a verificação.

Ciclo de vida da obra (Job.status)

Estados em base44/entities/Job.jsonc. Transições disparadas por employer/worker nas páginas, com PINs (presença diária e conclusão) durante in_progress. A conclusão final passa por completeJob após ambas as avaliações.

Jornada ponta a ponta (worker ↔ employer)

Sequência completa: criação do anúncio, descoberta no mapa, candidatura/proposta (gera chat + notificação + XP), aceitação, PINs, conclusão e avaliação cega (visível quando ambos avaliam ou após 7 dias).

Modelo de dados (entidades & relações)

Entidades Base44 em base44/entities/*.jsonc. User é gerido pela autenticação (não criável em massa pela API de dados) e espelhado para a tabela users do Supabase.

Arquitetura & backend (Base44 ↔ Supabase)

O cliente React fala com o Base44 (entities + funções Deno). Lógica sensível (avaliações, XP, estados) vive nas funções. Dados são espelhados para o Supabase via funções de sync (webhook/automação). Autorização usa role — nunca user_type; os PINs são apenas UX (gerados no cliente).

Admin & moderação

O AdminDashboard tem visão agregada e ferramentas de moderação. Ratings ≤ 2 geram alerta. A Blacklist regista motivo/severidade/evidência e altera o status do utilizador.
